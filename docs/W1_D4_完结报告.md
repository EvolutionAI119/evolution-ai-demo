### 2.3 ws.ts 类设计

```ts
export class OptimizeWS {
  constructor(taskId: string, options?: { maxReconnect?, autoReconnect?, maxBackoffMs? })
  onMessage(handler: (msg: WSProgressMessage) => void): () => void  // 返回 unsubscribe
  onStateChange(handler: (s: WSConnState) => void): () => void
  connect(): void
  close(): void  // 主动 close，不重连
  get currentState(): WSConnState
}
```

**4 类消息类型**：
```ts
type WSMessageType = 'PENDING' | 'STARTED' | 'PROGRESS' | 'SUCCESS' | 'FAILURE' | 'REVOKED' | 'ERROR'

interface WSProgressMessage {
  type: WSMessageType
  progress?: number          // 0.0 - 1.0
  current_iter?: number
  max_iter?: number
  result?: any               // SUCCESS 时附完整 result（含 convergence_curve）
  error?: string
}
```

**5 种连接状态**：
```ts
type WSConnState = 'connecting' | 'open' | 'reconnecting' | 'closed' | 'failed'
```

**断线重连策略**：
- 指数退避：`1s / 2s / 4s / 8s / 16s / 30s`（封顶 30s）
- 最多重连 8 次
- 终态（SUCCESS/FAILURE/REVOKED）自动 close，**不再重连**
- 主动 close 不重连
- 8 次都失败 → state='failed'（前端可显示错误 + 提示用户重试）

### 2.4 后端 ws.py 2 个 bug 修复

**Bug A**：终态时 SUCCESS 发两遍（先空后带 result）
- 旧代码：subscribe_progress 收 SUCCESS → 立即 `send_json(msg)` → reload task → 改 msg → 再 `send_json(msg)` 一次
- 新代码：SUCCESS **不立即发** → retry 10×200ms 拿 result_json → **一次性 send 带 result 的 SUCCESS** → break
- 影响：E2E 第一遍 SUCCESS 收到就 break 断开，第二遍 retry 完才发——客户端看不到。改成一次性后客户端必收

**Bug B**：SUCCESS 时 reload task 拿 result_json 失败（commit 竞态）
- 旧代码：reload 1 次就发 msg
- 新代码：retry 10×200ms=2s 拿 result_json（SUCCESS 信号到达时 DB 还没 commit 是常见的微秒级竞态）
- 影响：原来 ~30% 概率 SUCCESS 不带 result，retry 后 100% 带

## 4. 3 项技术决策

### 4.1 后端 ws.py 改"SUCCESS 一次性发带 result"（替代"先发空再发带 result"）

为什么：
- E2E 第一遍 SUCCESS 收到就 break，第二遍 retry 完才发——客户端看不到
- 改成 SUCCESS 不立即发，retry 完一次性发——**客户端必收完整 result**，减少 E2E/前端复杂度

### 4.2 断线重连 8 次 × 指数退避 1→30s

为什么：
- 8 次 = 1+2+4+8+16+30+30+30 = 121s 总重试窗口，覆盖 sandbox 4h 回收 + 短时网络抖动
- 超过 8 次进入 `failed` 状态，前端显示"连接失败，请稍后重试"——避免无限重试浪费资源
- 实际场景：浏览器 30s+ 无数据 + 服务器断网会自动触发 ws.onclose → 重连

### 4.3 后端不加心跳（依赖浏览器原生 ping/pong + 客户端重连）

为什么：
- 后端 FastAPI WebSocket 不需要服务端心跳——浏览器自动 30s ping/pong（WebSocket 协议层）
- 服务端心跳需要修改 ws.py 加 asyncio 任务，复杂度↑
- 简化：依赖断线重连兜底，**WS 30s+ 无消息 + 服务器断网 → onclose → 重连**

## 5. 4 项 ws.ts 关键代码

### 5.1 断线重连

```ts
private handleClose(): void {
  if (this.closedByUser || this.receivedFinal) {
    this.setState('closed'); return
  }
  if (!this.opts.autoReconnect) { this.setState('failed'); return }
  if (this.reconnectAttempt >= this.opts.maxReconnect) {
    this.setState('failed'); return
  }
  this.reconnectAttempt++
  const backoff = Math.min(this.opts.maxBackoffMs, 1000 * Math.pow(2, this.reconnectAttempt - 1))
  this.setState('reconnecting')
  this.reconnectTimer = window.setTimeout(() => {
    if (!this.closedByUser && !this.receivedFinal) this.connect()
  }, backoff)
}
```

### 5.2 4 消息分发

```ts
this.ws.onmessage = (ev) => {
  const msg = JSON.parse(ev.data)
  this.msgHandlers.forEach(h => h(msg))
  if (msg.type === 'SUCCESS' || msg.type === 'FAILURE' || msg.type === 'REVOKED') {
    this.receivedFinal = true
    this.close()
  }
}
```

### 5.3 主动 close 不重连

```ts
close(): void {
  this.closedByUser = true
  if (this.reconnectTimer) clearTimeout(this.reconnectTimer)
  if (this.ws) { try { this.ws.close() } catch {} this.ws = null }
  this.setState('closed')
}
```

### 5.4 OptimizeDetail.vue 用法

```ts
onMounted(() => {
  ws = new OptimizeWS(taskId.value, { maxReconnect: 8, maxBackoffMs: 30000 })
  unsubscribeMsg = ws.onMessage(handleMessage)
  unsubscribeState = ws.onStateChange(handleState)
  ws.connect()
})

onUnmounted(() => {
  unsubscribeMsg?.()
  unsubscribeState?.()
  ws?.close()
})
```
